# Claude Code 新手常见错误和修正方法

Claude Code 很适合辅助阅读代码和生成修改，但新手最容易犯的错误是跳过“工程检查”。看起来像是工具帮你省时间，实际上如果不读上下文、不看 diff、不跑检查，后面会花更多时间解释为什么改坏了别的地方。

下面按常见错误拆开。你可以同时参考 [Claude Code 新手使用检查清单](/blog/claude-code-beginner-mistakes-checklist)、[Bug 修复交付说明模板](/blog/bugfix-delivery-note-template) 和 [/tools/error-explainer](/tools/error-explainer)。

## 适合谁

适合已经能让 Claude Code 改出代码，但不确定如何判断质量的新手。你可能遇到过“模型说修好了，但项目跑不起来”的情况。

也适合准备接代码小单的人。客户通常不关心你用了什么 AI 工具，只关心问题是否解决、是否有测试证据、是否没有引入新风险。

## 不适合谁

不适合高风险生产任务。客户要求你处理真实付款、数据库、密钥、隐私数据或安全漏洞时，不要把它当成普通练习。

也不适合完全不懂项目运行方式的人。如果你连项目怎么启动、怎么构建、怎么回滚都不知道，先做学习和诊断，不要直接承诺修复。

## 具体步骤

错误一：不读上下文就要求工具修复。修正方法是先让工具总结目录、相关文件、错误路径和现有约定。你要先理解任务在哪个模块发生。

错误二：接受大范围改动。修正方法是要求最小改动，并在 diff 里检查是否出现无关文件、无关重构、格式化大面积变化或依赖升级。

错误三：不运行检查。修正方法是根据项目跑对应命令。能跑 `lint` 就跑 `lint`，能跑 `build` 就跑 `build`，能手动验证页面就保留截图或步骤。

错误四：复制客户敏感信息。修正方法是脱敏。日志里只保留错误码、路径和必要上下文；密钥、账号、订单、邮箱、合同内容都要替换成占位符。

错误五：交付说明太空。修正方法是写出原问题、修改点、验证结果、未覆盖范围。客户以后回看时，能知道这次到底完成了什么。

## 风险提醒

不要让模型替你判断平台规则和客户授权。工具可能建议“直接修改配置”或“查看后台”，但是否能做取决于客户授权、平台规则和风险等级。

不要把模型输出当成事实。错误原因、依赖版本、API 行为、部署规则都可能被说错。关键结论必须通过代码、文档或实际命令验证。

## 复盘方法

每次出错都值得复盘：是需求没问清，还是工具改动太大，还是自己没看 diff，还是测试没覆盖？把原因写下来，下一次就能减少重复踩坑。

如果你发现同类错误反复出现，可以把检查点做成固定模板。例如“修改前状态、修改后 diff、运行命令、交付说明”。模板越稳定，越不容易被工具输出带偏。

## 交付说明示例

```text
本次处理范围：
- 修复：
- 修改文件：
- 验证方式：
- 未覆盖：
- 客户需要确认：
```

这个模板的重点是让客户知道边界。比如你修了测试环境里的构建失败，就不要默认承担生产部署；你解释了 API 报错原因，就不要默认代管客户账号；你调整了一个组件，就不要默认重构整个页面。

## 新手更稳的工作方式

每次只让 Claude Code 做一个明确动作：总结文件、解释错误、提出修改方案、执行小改动、整理交付说明。不要一次性要求它“完整修复并优化所有问题”。任务拆得越小，你越容易判断输出质量。

如果你看不懂工具给出的修改，就不要交付给客户。可以把这次任务降级为学习记录，或请有经验的人复核。新手项目练习的底线是：能解释，能验证，能承担本次范围内的结果。

## 最小改动原则

新手最稳的默认策略是最小改动。客户说按钮失效，就先定位按钮、事件、接口和错误，不要顺手重写整个页面。客户说构建失败，就先看报错、依赖和配置，不要马上升级一堆包。

最小改动不是保守拖延，而是让你更容易证明因果关系。改动越小，越容易说清“为什么这样改”；改动越大，越难判断到底是哪一步解决了问题，也更难回滚。

## 检查顺序

建议固定顺序：先看客户原始需求，再看仓库状态，再改目标文件，再看 diff，再跑检查，最后写交付说明。不要把顺序倒过来。先写交付说明、后补测试证据，很容易漏掉真实问题。

如果检查失败，也要记录失败原因。失败不是坏事，它说明你还在验证，而不是盲目交付。客户看到清楚的失败记录和下一步建议，通常比看到一句含糊的“还在处理中”更容易理解。
