# Tailwind 样式不生效怎么检查：接单前怎么判断能不能做

Tailwind 样式修复看起来像小单，但范围可能差很多。有些只是 class 写错，有些是配置没有扫描组件，有些是组件库覆盖，还有些是生产构建问题。接单前不判断范围，很容易把诊断工作误报成简单修改。

这篇仍是草稿，发布前需要人工补充真实报价案例和平台规则核对。

## 适合谁

适合准备接 CSS、Tailwind、Next.js 页面小修任务的新手。你已经能本地运行项目，并能用浏览器开发者工具查看元素和样式。

也适合正在写 Proposal 的人。你需要把“我能先诊断”说清楚，而不是一开始承诺完整修复。

## 不适合谁

不适合没有仓库、没有截图、没有目标页面、没有期望效果的需求。也不适合客户要求你直接在生产环境里试改的任务。

如果客户项目包含未授权模板、付费设计素材、后台账号或敏感数据，需要先确认权限和边界。

## 接单前先问什么

先问五件事：当前页面链接或截图、期望效果、项目技术栈、能否提供测试环境、是否有最近改动记录。这些信息决定你能不能判断问题类型。

如果客户能提供仓库和本地运行命令，任务通常可以先做诊断。如果客户只给一张模糊截图，你只能给可能原因，不能确认修复范围。

## 判断能不能做

可以做的情况：问题只影响少数页面，客户有明确截图和期望效果，项目能本地运行，修改后能截图验收。

需要先诊断的情况：样式问题影响多个页面，客户不确定最近改了什么，或者本地和线上表现不同。

应该暂缓的情况：客户要求直接改生产站、不给备份、不提供访问方式、要求你处理未授权素材，或者任务牵涉到账户权限。

## 具体步骤

1. 保存客户原始描述和截图。
2. 询问项目框架、运行命令和目标页面。
3. 先做本地安装和构建检查。
4. 用开发者工具确认 class 是否出现在目标元素上。
5. 判断问题属于写法、配置、覆盖、缓存还是生产环境。
6. 把诊断结果发给客户确认。
7. 确认范围后再进入修复报价。

## 可复制 Proposal 片段

```text
I can start with a Tailwind style diagnosis before confirming the final fix. I will check the page, expected design, Tailwind setup, class rendering, CSS output, and possible overrides. After that I can confirm whether this is a small style fix or a configuration/build issue.
```

## 报价边界

小范围 class 修复可以按页面或组件报价。配置、构建和多页面样式异常，建议先按诊断报价。诊断交付物可以包括问题分类、截图、修改建议和风险说明。

不要把“看一下”当免费完整修复。接单时要写明是否包含生产发布、响应式检查、设计还原、浏览器兼容和后续维护。

## 验收边界

样式任务的验收标准要尽量可截图。比如“移动端按钮间距恢复”“首页卡片背景色按设计稿显示”“导航 hover 状态生效”。不要写成“页面更好看”，这种标准很难判断是否完成。

如果客户希望你“顺便优化一下”，要把它拆成新的范围。优化可能包括配色、布局、响应式、动效和组件重构，不应和一个 Tailwind 报错修复混在一起。

交付时附上修改文件、截图、运行命令和未覆盖范围。这样即使客户后续继续找别人维护，也能看懂你做了什么。

如果客户需要你发布到线上，要单独确认部署权限和回滚方式。样式修复本身和生产发布不是同一件事，不要在报价里默认包含。

如果只交付代码修改，也要说明上线由谁负责。

这句话能把开发交付和部署责任分开，减少后续争议。

## 风险提醒

不要承诺所有页面一次修好。不要在没有授权时改客户仓库或生产环境。不要公开客户截图和代码。不要把诊断阶段包装成已完成修复。
