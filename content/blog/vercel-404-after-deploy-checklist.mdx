# Vercel 部署后 404 检查清单：逐页验收更稳

Vercel 部署后 404，最怕只凭一个页面下结论。首页打不开、某个详情页打不开、自定义域名打不开、静态资源打不开，对应的排查方向都不一样。检查清单的目标是把“页面打不开”变成一组可验证的结果。

这篇草稿可以作为上线验收表使用。想先理解原因，可以读 [Vercel 部署成功但页面 404](/blog/vercel-404-after-deploy)。

## 适合谁

适合已经完成一次部署、需要确认页面是否能访问的新手。你可以把这篇当成客户交付前的自检表，避免只看 Vercel 后台状态就交付。

也适合做项目时做验收记录。客户要的是可访问的网站，不是一个后台截图。逐页记录能让沟通更清楚。

## 不适合谁

不适合仍停留在构建失败阶段的人。如果构建还没通过，404 检查没有意义。先回到 [build failed 排查清单](/blog/vercel-build-failed-causes-checklist)。

如果你没有项目权限，也没有任何预期 URL，只能做非常有限的外部观察。不要把外部观察说成完整诊断。

## 清单一：确认访问对象

先写清楚你正在测试哪个链接：Vercel 预览链接、生产部署链接，还是自定义域名。很多沟通误会来自链接不一致。客户说“打不开”，你打开的是另一个部署，自然会得出不同结论。

记录时建议写成表格：URL、环境、预期页面、实际状态、截图、测试时间。每一行只测试一个页面。

## 清单二：测试首页和关键路由

首页只是第一步。还要测试导航里最重要的页面，例如关于页、价格页、文章页、登录页、后台页、表单页。每个页面都要从浏览器直接打开一次，再从站内链接点击一次。

如果直接打开 404，但站内点击正常，可能涉及客户端路由或静态导出配置。如果站内点击也失败，可能是路由文件、链接地址或构建输出的问题。

## 清单三：测试动态路径

动态页面要至少测试三个样本：一个已知存在的 slug，一个大小写不同的 slug，一个不存在的 slug。这样能判断是所有动态页都坏，还是某些数据没生成。

如果项目有 CMS 或数据库，记录数据来源是否可用。动态路径 404 很多时候不是 Vercel 本身问题，而是构建时没有拿到对应内容。

再补一个容易漏掉的点：如果文章、商品或案例页面来自外部数据源，要确认构建时使用的是预览数据还是生产数据。有些项目本地能看到内容，是因为本地 `.env.local` 指向测试库；线上没有同样配置，就会生成不出对应页面。检查表里可以加一列“数据来源”，写清楚这个 URL 依赖哪个接口、哪张表或哪个 CMS 集合。

## 清单四：检查静态资源

页面能打开但样式、图片或脚本 404，也属于上线失败。检查浏览器 Network 面板里是否有资源路径错误、大小写错误、`/public` 路径误用或 CDN 缓存问题。

新手尤其要注意文件名大小写。本地 Windows 环境可能不敏感，线上环境可能更严格。一个字母大小写不同，就会导致图片或模块找不到。

## 清单五：检查项目配置

确认 Root Directory、Build Command、Output Directory 和框架识别是否匹配项目。静态站、Next.js、Vite、Astro 的输出方式不同，不能套用同一个目录。

如果仓库是 monorepo，要额外确认构建的是哪个应用。部署成功但打开空页面或 404，经常是部署到了错误子目录。

配置检查还要写明“谁确认”。如果是你自己练习，可以写自己的名字；如果是客户项目，就让客户确认生产域名、环境变量和项目归属。上线问题经常不是技术不会，而是责任边界没人写下来，后面所有人都以为别人已经确认过。

## 风险提醒

逐页验收不只是技术动作，也是交付保护。没有验收记录，客户后续说某个页面一直打不开，你很难说明当时状态。每次上线都要保留预览链接、截图和测试时间。

涉及自定义域名时，不要频繁修改 DNS。域名解析可能需要等待，也可能指向了别的项目。修改前先记录当前状态。

## 具体步骤

第一步，列出所有必须访问的 URL。

第二步，分别测试 Vercel 预览链接和自定义域名。

第三步，检查首页、关键路由、动态页面、静态资源和移动端。

第四步，把异常分成路由、输出目录、根目录、域名和数据来源五类。

第五步，把结果交给客户确认。需要下载验收表或联系人工协助，可以从 [工具导航](/tools) 进入。
