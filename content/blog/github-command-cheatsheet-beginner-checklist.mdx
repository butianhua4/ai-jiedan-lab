# GitHub 常用命令新手检查清单

这份清单用于练习 Git 和 GitHub 常用命令。完整解释见 [GitHub 常用命令表新手怎么记](/blog/github-command-cheatsheet-beginner)。你可以在自己的练习仓库里逐项填写，不要直接拿客户仓库试命令。

目标很简单：每次提交前知道自己在哪个分支、改了哪些文件、准备提交什么、是否有权限推送。

## 适合谁

适合正在用 GitHub 管理个人练习项目、作品集网页、内容草稿或 AI 辅助代码修改的新手。你可以用它建立固定流程，减少误提交。

也适合准备接网页小改、报错修复、文档整理等低风险任务的人。客户项目里，命令熟练度和记录习惯都很重要。

## 不适合谁

不适合在客户生产仓库里第一次练命令。真实协作仓库可能有分支保护、CI、代码审核和发布流程，误操作成本高。

也不适合学习高风险命令。涉及强制推送、重写历史、批量删除、清理工作区的命令，新手先不要碰，除非有人明确指导并确认影响范围。

## 风险提醒

提交前必须检查是否包含密钥、客户隐私、`.env` 文件、后台截图、账号信息或无关改动。AI 工具改完文件后尤其要看 diff。

如果需要客户提供仓库权限、添加协作者、批准 PR、修改保护规则或触发部署，把它写成待确认事项。没有授权时，不要把远程推送写成已完成。

## 具体步骤

按下面顺序练习：

1. 确认当前仓库
2. 查看当前分支
3. 查看文件状态
4. 查看具体差异
5. 暂存目标文件
6. 提交清楚消息
7. 推送到自己的远程仓库
8. 在 GitHub 页面核对提交
9. 写入复盘记录

如果任何一步看不懂，先停下来查清楚，不要继续复制下一条命令。

## 仓库检查

- 当前目录是不是目标项目？
- 是否已经是 Git 仓库？
- 当前远程仓库是谁的？
- 是否是自己的练习仓库？
- 是否需要客户授权？

可以用：

```bash
git status
git remote -v
```

看到陌生远程地址时，不要急着推送。先确认这是你的仓库还是客户/团队仓库。

## 分支检查

- 当前分支：
- 是否允许直接提交：
- 是否需要新建分支：
- 分支名称是否能说明任务：
- 是否需要 PR：

常见命令：

```bash
git branch
git switch -c fix-small-layout
```

如果项目要求从 issue 或任务号创建分支，按项目要求来。

## 修改检查

- 哪些文件修改了？
- 是否都是本次任务相关？
- 是否有自动格式化造成的大范围 diff？
- 是否有 AI 工具改到无关文件？
- 是否有临时文件或截图误加入？

常见命令：

```bash
git status
git diff
```

不要只看状态列表。至少抽查关键 diff，确认内容合理。

## 暂存检查

新手尽量不要一上来用 `git add .`。更稳的是逐个添加目标文件：

```bash
git add path/to/file.md
```

如果确实要添加多个文件，先确认每个文件都属于本次任务。暂存后再运行一次：

```bash
git status
```

确认 staged 文件没有多余内容。

## 提交检查

- 提交消息是否清楚？
- 是否说明了本次改动？
- 是否避免了 `final`、`fix`、`update` 这种模糊词？
- 是否只包含一个主题？

示例：

```bash
git commit -m "Fix mobile layout overflow"
```

如果一次提交包含多个无关主题，下次回滚和复盘都会很麻烦。

## 推送检查

推送前确认：

- 远程仓库是自己的还是客户的？
- 当前分支是否正确？
- 是否有权限？
- 是否需要先 pull？
- 是否会触发部署或 CI？

常见命令：

```bash
git push
```

如果这是客户仓库，推送前先确认协作流程。没有权限或授权不清楚时，写成待确认事项。

## 冲突和暂停检查

如果拉取或推送时出现 conflict、rejected、non-fast-forward 这类提示，不要急着复制网上的清理命令。先保存完整报错，运行 `git status`，确认哪些文件冲突、哪些文件是你改的、哪些文件来自远程。

新手在客户项目里遇到冲突时，最稳的做法是暂停并说明情况：本地改动和远程更新发生冲突，需要确认保留哪一版。不要为了让命令通过而删除文件、重写历史或强行推送。

自己的练习仓库可以用来学习冲突处理，但客户仓库要先走沟通流程。需要项目负责人确认的文件，写成明日待办；确认后再继续处理。

## 复盘记录

每次练习后写三行：

```text
我运行了什么命令：
我看到什么结果：
下次要注意什么：
```

这比单纯收藏命令表更有效。你会慢慢知道每条命令在什么场景下使用。
