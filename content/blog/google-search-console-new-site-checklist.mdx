# Google Search Console 新站检查清单

这份清单用于新站准备使用 Google Search Console 前的安全自查。完整说明见 [Google Search Console 新站怎么用](/blog/google-search-console-new-site)。它把检查分成两类：公开可做的检查，以及需要授权后才能做的后台操作。

不要跳过权限确认。只要涉及登录、验证站点、改 DNS、上传文件、添加 meta 标签、查看搜索数据或分配权限，都先写入明日待办。

## 适合谁

适合刚上线个人网站、作品集、工具页、模板页或客户小站的新手。你想先确认公开基础是否正常，再决定是否进入 GSC。

也适合做 AI 项目练习项目。没有客户授权时，只做公开检查并记录限制。

## 不适合谁

不适合没有站点所有权、没有客户确认、无法说明验证方式的情况。此时不要进入后台操作。

也不适合复杂技术 SEO、排名恢复、惩罚排查、企业级权限管理或大规模抓取问题。这些需要更多经验和权限。

## 风险提醒

不要把 Search Console 当成普通插件。验证站点后，你可能能看到搜索数据、提交 sitemap、查看覆盖报告和管理权限。未经授权操作会带来信任和合规风险。

不要承诺收录、排名或流量。GSC 是观察和诊断工具，不是搜索结果保证。

## 具体步骤

按顺序检查：

1. 打开首页和重要页面。
2. 检查标题、描述、canonical 和 noindex。
3. 打开 sitemap.xml。
4. 打开 robots.txt。
5. 确认草稿没有进入 sitemap。
6. 记录公开检查截图。
7. 确认站点所有权。
8. 确认验证方式和权限边界。
9. 把后台操作写入明日待办。

前六步可以先做；后三步需要授权。

## GSC 后台待办

需要授权后再做：

- 登录 Search Console
- 创建或选择 property
- 完成站点验证
- 提交 sitemap
- 查看页面索引报告
- 查看搜索查询
- 查看点击和展示数据
- 查看问题提示
- 添加或移除用户权限

这些事项不要写成已经完成，除非有真实截图和授权记录。

## 交付记录

如果是客户项目，至少交付：

- 公开检查结果
- sitemap 和 robots 状态
- 是否发现草稿泄漏
- 是否发现 noindex 问题
- 需要授权的待办清单
- 建议的验证方式
- 风险说明

这样即使没有后台权限，也能提供清楚价值。

## 复盘和证据保存

检查完成后，把证据放到固定位置。建议每个站点保存一份简短记录：检查日期、站点 URL、检查页面、sitemap 截图、robots 截图、发现的问题、没有权限查看的内容、下一步待办。这样下一次继续处理时，不需要从聊天记录里重新翻。

如果你没有 GSC 后台权限，也要明确写出限制。例如“本次只完成公开页面检查，未登录 Search Console，未验证站点所有权，未查看搜索查询和点击数据”。这不是扣分，而是范围清楚。对新手来说，诚实说明没做什么，比把公开检查包装成完整 SEO 审计更可靠。

客户项目还要额外保存授权记录。授权可以是平台消息、邮件确认或项目说明，但不要只靠口头印象。需要修改 DNS、上传文件、添加 meta 标签、分配权限时，必须先把授权和回滚方式记录下来，再进入操作。

## 明日待办

统一记录：

- 获取客户书面授权
- 确认站点所有权
- 登录 GSC
- 选择验证方式
- 修改 DNS
- 上传验证文件
- 添加 meta 标签
- 提交 sitemap
- 保存后台截图
- 分配或移除权限

待办没有完成前，方案保持草稿。

## 最终核对

发布或执行前问自己：

- 是否只做了公开可验证检查？
- 是否没有擅自验证站点？
- 是否没有修改 DNS 或生产文件？
- 是否没有承诺收录和排名？
- 是否记录了授权待办？
- 是否确认 draft 不进入 sitemap？
- 是否需要人工复核？

如果答案不清楚，就暂停。
