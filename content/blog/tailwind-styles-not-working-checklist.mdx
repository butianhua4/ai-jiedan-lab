# Tailwind 样式不生效怎么检查：新手检查清单

Tailwind 样式问题适合用清单排查，因为它经常不是一个点造成的。你需要同时看页面、组件、配置、构建和浏览器结果。

这篇仍是草稿，发布前需要补充真实案例和版本核对。

## 适合谁

适合能打开项目、能运行本地开发服务、能查看开发者工具的新手。你不需要一开始就知道答案，但需要愿意逐项排除。

也适合准备用 AI 辅助接 CSS 小单的人。清单能帮助你判断任务是否边界清楚，是否需要先做诊断。

## 不适合谁

不适合没有仓库、没有页面链接、没有截图、没有运行方式的需求。没有证据时，清单只能记录缺口，不能直接给修复结论。

如果客户要求你直接改生产站或改付费主题源码，需要先确认授权和备份。

## 检查清单

逐项确认：

1. 页面路径和截图是否保存。
2. 期望效果是否写清楚。
3. 项目框架是否确认。
4. Tailwind 版本是否确认。
5. 全局 CSS 是否被入口文件引入。
6. 扫描路径是否包含当前组件目录。
7. 目标元素是否带有对应 class。
8. class 是否拼写正确。
9. 是否使用动态拼接 class。
10. 是否被组件库、全局 CSS 或内联样式覆盖。
11. 是否重启开发服务。
12. 生产构建是否通过。
13. 是否有回滚方式。

## 具体步骤

1. 先只改一个明显的测试类，例如背景色或边框。
2. 看浏览器元素上是否出现这个类。
3. 如果类没有出现，检查组件渲染和条件判断。
4. 如果类出现但样式没有生效，检查生成 CSS 和覆盖关系。
5. 如果本地有效但线上无效，检查构建流程、部署缓存和生产环境路径。
6. 每次只验证一个假设，避免同时改配置和组件。

## 清单怎么转成客户回复

如果客户材料完整，可以回复：“我可以先做一轮 Tailwind 样式诊断，检查入口 CSS、扫描路径、组件 class、样式覆盖和构建结果。诊断后再确认修复范围。”

如果客户材料不完整，可以回复：“请先提供页面链接、截图、仓库或可复现步骤。没有这些信息时，我只能判断可能原因，不能确认修复范围。”

这样的回复比直接说“我能修”更稳，也能避免把配置问题误报成简单样式修改。

## 复查优先级

如果时间有限，先查最能缩小范围的项目：目标元素是否出现 class、生成 CSS 是否包含规则、是否存在覆盖样式。这三项能把问题分成组件渲染、构建生成和样式优先级三类。

然后再查配置文件、目录路径和开发服务重启。很多新手一上来就改配置，但其实元素根本没有渲染出来。先看浏览器里的真实 DOM，能少走很多弯路。

最后检查线上环境。本地修好不代表线上已经更新，部署平台可能还在使用旧构建。把本地截图和线上截图分开保存，方便客户验收。

如果你准备把排查写成作品集，也要保留“错误路径”。说明你先排除了哪些可能性，再找到真正原因。这样的记录比只展示最终截图更可信。

同时标出哪些文件没有改。客户最关心的是影响范围，清楚说明未触碰的目录，可以减少后续误会。

如果改动影响多个组件，建议单独列一个“已验证页面”清单，避免客户默认全站都已经检查。

## 风险提醒

不要在没有备份时修改全局样式。不要承诺一次修复所有页面。不要把客户付费模板或未授权素材公开。不要在生产环境边试边改。
