# 向量数据库怎么选：新手先理解 embedding 和检索

向量数据库不是普通新手每天都会用的工具，但在 AI 知识库、RAG、语义搜索和 Agent 记忆里非常常见。它的作用可以简单理解为：把文本、图片或其他内容转成向量，再根据相似度找到相关内容。用户问一个问题，系统不只是按关键词查，而是找语义上接近的资料。

这篇是 AI 基建草稿，发布前需要核对具体产品和价格。RAG 流程可以看 [RAG 知识库怎么搭](/blog/rag-knowledge-base-beginner-guide)。

## 适合谁

适合想做知识库问答、语义搜索、文档推荐、相似案例检索或 Agent 长期记忆的人。你不一定要马上选复杂产品，但要理解向量库解决什么问题。

也适合接 AI 项目评估的人。客户说要知识库，你要知道是否真的需要向量数据库，还是普通全文搜索和结构化数据库就够。

## 不适合谁

不适合为了赶潮流而强行使用向量库的人。如果资料很少、问题很简单、关键词匹配就能解决，向量库可能增加复杂度。

如果项目涉及严格权限、企业内部资料或大量数据，向量库选择还要考虑安全、备份、审计和运维。

## embedding 是什么

embedding 是把文本转成一串数字向量的过程。语义相近的文本，向量距离通常更近。比如“如何部署大模型”和“LLM 怎么上线”可能在语义上接近。

向量库保存这些向量和原始片段的引用。检索时，系统把用户问题也转成向量，再找最相近的内容。

## 向量库解决什么问题

关键词搜索适合查明确词语，向量检索适合查语义相关内容。比如用户问“客户不满意怎么处理”，文档里可能写的是“售后投诉流程”，关键词不完全一样，但语义相关。

RAG 系统常用向量库来找资料，再让模型基于资料回答。没有检索，模型只能靠自身知识和上下文，容易漏掉业务文档。

## 选择维度一：数据规模

资料只有几十篇文档时，简单方案可能足够。资料很多、更新频繁、多人使用时，就要考虑性能、索引更新和成本。

不要一开始就按最大规模设计。先估算文档数量、片段数量、查询频率和增长速度。

## 选择维度二：过滤和权限

真实项目里，用户不一定能看所有资料。向量检索需要支持按部门、项目、角色、时间、文档类型过滤。

如果只做相似度检索，不做权限过滤，可能把不该出现的资料带进回答。企业知识库尤其要注意。

权限过滤最好从一开始就设计，不要等上线后再补。向量库里通常会存文档片段和元数据，元数据可以包括项目、部门、文档类型、更新时间和可见范围。没有元数据，后面想做权限和筛选会很痛苦。

## 选择维度三：成本和维护

向量库可能有存储成本、查询成本、运维成本和备份成本。embedding 本身也可能消耗费用或计算资源。

新手评估时不要只看工具是否热门，要看团队是否能维护、是否有文档、是否方便导出和迁移。

还要看数据更新方式。知识库资料如果每天变化，向量索引也要能增量更新；如果资料很少变化，可以用更简单的批量更新。更新流程不清楚时，系统刚上线可能好用，几周后就会因为旧资料越来越多而变差。

## 风险提醒

向量检索不是总能找到正确资料。切分不好、embedding 不适合、过滤条件错误、资料过期，都会影响结果。

另一个风险是把检索结果直接当事实。RAG 回答仍然需要引用、置信度和人工复核，尤其是对外服务。

## 具体步骤

第一步，确认你要做关键词搜索、语义搜索，还是 RAG 知识库。

第二步，估算文档数量、片段数量、查询频率和权限需求。

第三步，选择 embedding 模型和向量存储方案。

第四步，测试检索效果：相关、无关、权限不足、找不到答案。

第五步，记录成本、维护人和更新流程。需要选择表或人工协助评估，可以从 [工具导航](/tools) 进入。
