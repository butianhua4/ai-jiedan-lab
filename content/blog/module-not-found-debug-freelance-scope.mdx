# module not found 接单前怎么判断能不能做

`module not found` 很适合做新手报错排查练习，但不是所有情况都适合直接接单。它可能只是路径大小写错误，也可能是客户仓库缺文件、私有包没有权限、构建环境和本地不一致。

如果你只想学习排查顺序，可以先看 [module not found 报错怎么定位文件](/blog/module-not-found-debug)。这一篇重点讲接单前怎么判断范围和风险。

## 适合谁

适合已经会看报错文件、会搜索文件名、会运行 build 命令的新手。你可以处理本地路径错误、明显导入错误、缺少普通公开依赖这类低风险问题。

也适合做作品集练习。你可以用自己的项目复现大小写错误、别名配置错误、缺依赖错误，再整理成案例。

## 不适合谁

不适合完全不了解项目结构、包管理器和构建命令的人直接处理客户仓库。模块缺失可能会牵出很多依赖。

也不适合客户不给仓库、不提供完整报错、只发一张红色截图就要求马上修好。材料不足时，可以先报价诊断，不要承诺修复。

## 风险提醒

接单时要防止把客户材料缺失当成自己的修复任务。比如客户没有提交某个组件、没有授权私有包、没有提供子模块或内部 registry 权限，这些需要客户补齐。

不要要求客户把私有包 token、仓库密码或内部源码发到聊天里。授权应该通过平台和仓库权限完成。

## 具体步骤

### 1. 接单前要材料

至少要客户提供：

- 完整报错。
- 触发命令。
- 相关仓库或最小复现。
- 包管理器。
- Node 版本。
- 是否涉及私有包。
- 是否本地和 CI 都失败。

没有这些材料时，只能做初步判断。

### 2. 判断可接范围

适合新手的范围包括：定位缺失模块、检查导入路径、修正大小写、补普通依赖、解释别名配置、写交付记录。

不建议新手独立处理的范围包括：私有包权限、复杂 monorepo、内部组件库迁移、构建系统重构、生产部署权限、客户未提交源码。

### 3. 报价怎么拆

可以拆成诊断和修复：

```text
Diagnosis:
- Identify the missing module and import location.
- Classify the cause as path, dependency, alias, environment, or permission.

Fix:
- Apply code or dependency changes only after the cause is confirmed.
- Private package access and missing client files must be confirmed by the repository owner.
```

这样能避免你把客户权限问题包进小修复报价里。

### 4. 明日待办怎么写

如果卡在客户材料，写成待办：等待客户补仓库文件、添加私有包权限、确认包管理器、提供 CI 日志、确认是否允许安装依赖。

待办要写清负责人。属于客户的不要写成你的未完成。

## 暂缓信号

遇到这些情况先暂缓：客户拒绝提供仓库或最小复现；报错涉及私有包但没有权限；项目是大型 monorepo；客户要求直接删除报错组件；你无法运行任何验证命令。

暂缓时可以交付诊断问题清单，而不是硬接修复。

## 适合做成作品集吗

`module not found` 很适合做成匿名作品集，因为它能展示你的排查顺序。作品集里可以展示缺失模块、导入文件、原因分类、修复前后验证结果，但不要展示客户私有路径、内部包名、仓库地址或完整日志。

如果最终原因是客户没有提供私有包权限，也可以写成案例：如何判断这不是代码路径错误，而是客户侧材料缺失。这类案例体现的是边界判断，不一定非要有代码改动。

公开前把所有路径和包名脱敏，保留问题类型即可。真实客户项目没有授权时，只做匿名复盘，不公开截图和源码。

## 交付边界怎么写

交付边界可以写成：本次先定位缺失模块和原因类型，普通路径或公开依赖问题可继续修复；私有包、客户漏文件、内部 registry、仓库权限问题需要客户确认后再继续。

这样写不会显得退缩，反而能让客户知道你在保护项目材料和访问权限。新手接单时，清楚边界比盲目承诺更可靠。
