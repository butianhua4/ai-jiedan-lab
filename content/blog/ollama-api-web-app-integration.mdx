# Ollama API 怎么接到网页应用：本地模型到前后端调用

Ollama 官方文档说明，安装后默认 API 会在本地 `http://localhost:11434/api` 提供服务，并支持通过 API 与模型交互。对新手来说，跑起 Ollama 只是第一步；要接到网页应用，还要考虑前端能不能直接访问、是否需要后端代理、请求超时、流式输出、错误处理和端口安全。

这篇是草稿，正式发布前需要核对 Ollama 最新 API 文档。先跑本地模型可以看 [Ollama 本地部署](/blog/ollama-local-model-deployment-guide)，做网页可以看 [AI 做网站](/blog/ai-website-builder-beginner-guide)。

## 适合谁

适合已经在本地运行 Ollama，并想把模型接进网页、桌面工具、内部小应用或实验项目的人。你可能已经能在命令行聊天，但还不知道前后端怎么调用。

也适合做本地 AI 工具原型的人。Ollama API 能让你把本地模型能力接到应用里，但上线边界要先说清楚。

## 不适合谁

不适合直接把本地 Ollama 服务暴露给公网的人。默认本地服务适合本机调用，不代表可以无权限开放给外部用户。

如果项目要给多人使用，尤其涉及客户数据和生产场景，需要做后端代理、访问控制、日志和安全评估。

## 第一步：确认 Ollama API 可用

先确认 Ollama 已经运行，模型已下载，并能通过本地 API 返回结果。官方文档给出的默认本地 API 地址是 `http://localhost:11434/api`。

测试时记录模型名称、请求内容、响应时间和错误情况。不要只测一个短问题，也要测长文本和异常输入。

## 第二步：不要让前端直接承担所有逻辑

简单 demo 里，前端可能直接请求本地 API。但更稳的做法是让网页前端调用自己的后端接口，再由后端去调用 Ollama。这样更容易处理权限、日志、超时和错误。

如果未来要部署到服务器，后端代理也更方便切换模型服务或加访问控制。

## 第三步：处理超时和响应速度

本地模型响应速度受机器、模型大小、上下文长度和输出长度影响。网页应用不能假设模型会很快返回。

要设置加载状态、超时提示、取消按钮和错误说明。用户等模型时，界面要让他知道系统仍在处理。

## 第四步：考虑流式输出

聊天类应用通常更适合流式输出，让用户边看边等。是否使用流式输出，要看你的前后端是否支持、错误处理是否清楚、日志是否能记录完整内容。

如果只是内部工具，先用普通响应跑通也可以。不要为了炫技一开始就做复杂流式流程。

## 第五步：权限和端口

本地开发时端口只给自己用。多人访问或服务器部署时，要考虑谁能调用接口、是否需要登录、是否要限制频率、是否保存输入输出。

不要把内网模型接口随便暴露到公网。尤其是模型能读取本地文件、调用工具或接入知识库时，风险会明显提高。

如果只是本地 demo，可以把浏览器、后端和 Ollama 都放在同一台机器上。只要准备给别人访问，就要重新设计网络边界：前端访问后端，后端访问 Ollama，外部用户不能直接打到模型服务端口。这个结构虽然多一步，但更容易控制权限和日志。

## 常见错误

常见错误是把浏览器直接接到本地模型接口，然后以为 demo 能跑就可以给别人用。只要用户环境、网络、模型端口或浏览器安全策略变化，这个方案就会出问题。更稳的做法是把模型服务封装在后端后面。

交付记录里要写清楚：Ollama 运行在哪台机器，网页应用运行在哪里，后端接口路径是什么，模型名称是什么，超时多久，失败时给用户什么提示。以后换模型或迁移服务器时，这些信息就是路线图。

## 风险提醒

Ollama API 很适合本地原型，但本地原型不等于生产系统。生产使用要考虑并发、监控、日志、权限、模型许可和备份。

另一个风险是前端泄漏。不要把内部接口、敏感 prompt、业务数据或不该公开的地址暴露给外部用户。

## 具体步骤

第一步，确认 Ollama 本地服务和目标模型能正常返回。

第二步，写一个后端接口代理 Ollama 请求。

第三步，前端调用自己的后端接口，处理加载、错误和取消。

第四步，测试短问题、长问题、异常输入、超时和多人使用。

第五步，整理接入记录。需要检查表或人工协助，可以从 [工具导航](/tools) 进入。
