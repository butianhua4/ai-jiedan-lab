# 报价邮件模板怎么表达不确定范围

客户需求不清楚时，不要为了显得专业就硬报最终价。更稳的报价邮件应该说清楚：哪些信息已经确认，哪些还不确定，第一步可以做什么，什么时候再给最终报价。

这篇仍是草稿。发布前需要人工补充真实邮件样例和平台沟通规则核对。

## 适合谁

适合收到模糊需求的新手，比如“帮我修网站”“页面有问题”“部署失败”“AI 工具能不能做”。这些需求不是不能接，而是不能直接报最终范围。

也适合用 AI 辅助写邮件的人。AI 可以润色语气，但不能替你确认范围和责任。

## 不适合谁

不适合客户已经明确要求正式合同、法律条款、复杂报价或企业采购流程的情况。这类邮件需要更正式的人工审核。

如果项目涉及生产系统、支付、用户数据、敏感账号或安全事项，不要只靠一封报价邮件推进。

## 邮件应该表达什么

一封谨慎的报价邮件要包括：

1. 复述你理解的需求。
2. 说明当前还缺哪些信息。
3. 提出第一步诊断或评估范围。
4. 写清最终报价需要哪些条件。
5. 列出不包含范围。
6. 请客户确认下一步。

这样写不是拖延，而是避免把未知问题包装成固定小单。

## 具体步骤

1. 先复制客户原始消息，标出已知和未知。
2. 列出你需要的截图、链接、仓库、日志、验收标准。
3. 把第一步定义为诊断，而不是完整修复。
4. 写出诊断交付物，例如原因判断、建议方案和下一步报价。
5. 避免承诺工期、结果和范围，除非材料已经足够。
6. 发出前检查是否要求客户提供不必要的敏感信息。

## 可复制邮件模板

```text
Hi,

Thanks for the details. Based on the current information, I understand the goal is:
- [your understanding]

Before I confirm the final quote, I need to review:
- screenshot or error log
- page link or repository
- expected result
- whether this is a test or production environment

I can start with a diagnostic pass first. The deliverable would be a short summary of the likely cause, recommended fix, and confirmed implementation scope.

Once the missing details are confirmed, I can provide a more accurate quote.
```

发送前要按客户项目改写，不要把模板原样复制。

## 中文表达版本

如果你用中文沟通，可以写：

```text
目前我能先确认需求方向，但还不能直接给最终报价。
为了避免范围误判，我需要先看页面链接、截图、报错日志和验收标准。
我建议先做一轮诊断，交付内容是问题判断、建议修复方案和后续报价范围。
确认这些信息后，再决定是否进入具体修复。
```

语气要稳，不要显得不懂；也不要装作已经确定原因。

## 什么时候可以给最终价

当客户已经提供复现步骤、仓库或链接、截图或日志、验收标准，并且权限边界清楚时，才适合给最终报价。否则更适合报诊断阶段，或者只说明“确认材料后报价”。

你也可以把报价拆成两个句子：“诊断阶段包含问题定位和修复建议；具体修复费用会在确认原因和改动范围后给出。”这样客户知道你不是回避报价，而是在保护双方对范围的理解。

## 报价邮件里的边界词

可以多用“diagnostic pass”“initial review”“confirmed scope”“after reviewing the details”这类表达。中文里可以写“初步诊断”“确认范围后报价”“需要先看材料”。这些词能让客户知道你会推进，但不会过度承诺。

要少用“肯定”“一定”“马上修好”“全包”这类词。尤其是你还没有看到仓库、日志或截图时，这些词会把你带进被动局面。

报价邮件也可以写“不包含范围”。比如“这一步不包含生产发布、账号配置和长期维护”。把不包含写清楚，不是不友好，而是避免客户把诊断理解成完整交付。

## 风险提醒

不要在材料不完整时承诺固定结果。不要要求客户发送密码、私钥、token 或后台账号。不要把诊断价写成完整修复价。

需要注册平台、加入仓库、查看预览链接、申请测试账号或客户授权的事项，放进明日待办。
