# Vercel 部署后 404 常见误区：别只盯着域名

部署后 404 很容易让人第一反应怪域名，但很多时候问题其实在路由、输出目录、根目录或动态数据。新手如果只盯着 DNS 改来改去，可能把原本简单的访问问题变成更难恢复的配置问题。

这篇草稿整理常见误区，帮助你在做教程和服务页时把风险说清楚。需要按表检查时，可以看 [Vercel 部署后 404 检查清单](/blog/vercel-404-after-deploy-checklist)。

## 适合谁

适合遇到“Vercel 已部署成功，但客户说打不开”的新手。你可能已经看到后台绿色状态，却不知道为什么页面仍然 404。

也适合准备交付网站的人。上线交付不是只发一个链接，而是要证明关键页面能打开，异常页面有解释，客户知道还需要确认什么。

## 不适合谁

不适合构建还失败的人。构建失败和访问 404 是两个阶段，先后顺序不能乱。

也不适合没有任何项目上下文的人。只凭一个 404 页面，最多能做外部判断，不能确认项目内部配置。

## 误区一：后台成功就算上线成功

Vercel 显示部署成功，只代表构建和发布流程结束。它不代表所有路由、资源、表单、移动端、SEO 文件和自定义域名都通过验收。

正确做法是把上线拆成两层：平台状态和用户访问状态。平台状态看部署是否 ready，用户访问状态看每个关键 URL 是否能打开。

## 误区二：只测试首页

首页正常不代表整站正常。文章详情、产品详情、登录页、后台页、表单页和 sitemap 都可能单独出问题。尤其是动态路由，必须用具体样本验证。

交付前至少测试三类页面：静态页面、动态页面、资源文件。记录 URL 和截图，后续沟通会省很多麻烦。

如果网站有导航菜单，还要从导航点击一次，再把同一个 URL 直接粘到浏览器打开一次。前者能检查站内链接，后者能检查刷新和直达访问。有些单页应用点击正常，但直接访问子路径 404，这种问题只看首页永远发现不了。

## 误区三：输出目录凭感觉填

不同框架的输出目录不同。Vite 常见是 `dist`，有些项目是 `build`，静态导出可能是 `out`，但不能只靠经验套。项目配置、构建日志和框架文档才是依据。

输出目录填错时，部署可能仍然完成，但发布出来不是正确网站。遇到这种情况，先看构建产物，再改 Vercel 设置。

## 误区四：把所有 404 都归因于域名

域名确实会导致访问问题，但不是唯一原因。如果 Vercel 预览链接也 404，优先查项目路由和输出；如果预览正常、自定义域名异常，再查 DNS 和域名绑定。

排查顺序很重要。先把预览链接跑通，再处理自定义域名，能减少大量无效修改。

## 误区五：不留验收记录

客户看到 404 会很焦虑，如果你没有记录，沟通会变成各说各话。验收记录至少包括测试时间、URL、状态、截图、修改项和待确认事项。

这些记录也能反过来成为内容资产。脱敏后可以整理成部署教程、排查清单和服务说明，帮助搜索流量找到你。

记录不是为了显得复杂，而是为了让下次问题更快定位。比如同一个页面昨天正常、今天 404，你可以立刻对比中间发生了什么：是否改了路由、是否更新了 CMS 内容、是否重新绑定域名、是否切换了部署分支。没有记录，就只能从头猜。

## 风险提醒

处理 404 时不要随意删除项目、改主域名、替换生产部署或清空配置。看似只是在修访问问题，实际可能影响线上业务。

如果客户网站已经对外使用，任何修改都要有授权、备份和回退方案。新手可以先做诊断，不要把高风险配置修改当成随手操作。

## 具体步骤

第一步，确认 build 是否成功，预览链接是否可访问。

第二步，分别测试首页、关键路由、动态路径和静态资源。

第三步，检查 Root Directory、Output Directory、路由文件和动态数据来源。

第四步，如果只有自定义域名异常，再检查域名绑定和 DNS 状态。

第五步，生成上线验收记录。需要下载模板或联系人工协助，可以通过 [工具导航](/tools) 找到入口。
