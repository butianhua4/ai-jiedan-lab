# Tailwind 样式不生效怎么检查

Tailwind 样式不生效时，先不要急着把 class 名改来改去。新手最容易犯的错，是把所有问题都当成“写错了类名”。真实项目里，问题可能来自全局 CSS 没引入、配置文件没有扫描到组件、构建缓存没有刷新、动态拼接 class 被忽略、组件路径写错，或者生产环境和本地环境不一致。

这篇仍是草稿，发布前需要人工核对不同 Tailwind 版本、框架配置和真实截图。

## 适合谁

适合正在调试个人项目、作品集、小型客户页面或练习站点的新手。你已经能打开项目，能运行安装和构建命令，也能查看浏览器开发者工具。

也适合准备接小型样式修复单的人。你需要先判断任务是简单样式问题，还是项目配置、构建系统、组件库冲突导致的问题。

## 不适合谁

不适合在客户生产站点上直接试错。也不适合没有仓库、没有复现方式、没有截图、没有本地运行权限的任务。

如果项目涉及大型设计系统、付费主题、未授权模板、生产发布或客户后台权限，需要先确认授权和回滚方式。

## 常见原因

第一类是入口文件问题。项目没有正确引入全局 CSS，或者 Tailwind 指令没有进入最终构建。第二类是扫描路径问题。组件放在 `app`、`pages`、`components`、`src` 或自定义目录里，但配置没有覆盖这些路径。

第三类是动态类名问题。例如把颜色、尺寸、间距拼接成字符串，Tailwind 可能无法在构建时识别。第四类是样式覆盖问题。组件库、全局 CSS、内联样式或更高优先级选择器可能把 Tailwind 类覆盖掉。

第五类是缓存和构建差异。本地开发服务没有重启、依赖版本改变、生产构建没有更新，都可能让你看到旧样式。

## 具体步骤

1. 保存截图、页面路径和期望效果。
2. 确认项目能本地运行，并记录使用的框架和 Tailwind 版本。
3. 检查全局 CSS 是否被入口文件引入。
4. 检查 Tailwind 配置或框架默认扫描路径是否覆盖当前组件目录。
5. 在浏览器开发者工具里看目标元素是否真的带有对应 class。
6. 如果元素有 class 但样式没出现，继续看生成的 CSS 和覆盖关系。
7. 如果元素没有 class，检查组件渲染条件、拼写和动态拼接。
8. 修改一处后重启开发服务或重新构建，再记录结果。

## 接单前怎么判断

如果客户能提供仓库、截图、目标页面和期望效果，这类任务可以先做诊断报价。如果客户只说“样式坏了”，但没有链接、截图或运行方式，就不要直接承诺修复。

你可以把任务拆成两段：先做一次样式诊断，交付问题定位和修复建议；确认问题范围后，再决定是否进入修复。这样比一开始就报固定价更稳。

## 修复后怎么验收

修复 Tailwind 样式后，不要只看当前浏览器里的一个页面。至少要保存修改前截图、修改后截图、运行命令和构建结果。如果任务涉及响应式样式，还要检查手机宽度和桌面宽度。

验收时可以写清楚三件事：修复了哪个元素、问题原因是什么、还有哪些页面没有覆盖。这样客户后续追加需求时，你能把新增内容和原范围分开。

如果只是诊断交付，也要给出明确结论。例如“问题更可能来自扫描路径”或“目标元素有 class，但被全局样式覆盖”。不要只写“我看了一下”。

## 风险提醒

不要直接修改客户生产环境。不要使用未授权模板素材。不要承诺所有页面都能在一次修复中完成。不要在没有回滚方式时改全局 CSS 或构建配置。
