# Vercel build failed 排查能不能接单：先看这 7 个边界

Vercel build failed 看起来像一个很适合新手的小任务：客户网站部署失败，你看日志、改配置、重新部署。但接单时真正要判断的不是“我会不会点 Vercel”，而是任务边界是否清楚、材料是否完整、风险是否可控、客户是否愿意配合验证。

这篇草稿把部署失败排查拆成 7 个接单边界。它不是报价承诺，而是帮你判断哪些任务适合做诊断，哪些任务应该暂缓。技术排查步骤可以参考 [Vercel build failed 常见原因](/blog/vercel-build-failed-causes)。

## 适合谁

适合已经会读基础日志、会看 `package.json`、能本地运行构建命令的新手。你不需要独立完成大型系统运维，但要能说清楚自己负责诊断、修复小范围配置，还是协助客户整理问题。

也适合正在搭建 AI 接单服务页的人。部署排查是很好的搜索入口，因为客户往往带着明确问题来找解决办法。文章、清单和询问表能帮助你把流量转成可判断的线索。

## 不适合谁

不适合完全看不懂日志、也不愿意按流程记录的人。部署失败不是只靠复制 AI 答案就能处理，尤其是客户项目里有真实业务时。

如果客户要求你在没有备份、没有授权、没有验收标准的情况下直接改生产环境，这类任务不适合新手。你可以建议先做诊断报告，而不是直接承诺完整修复。

## 边界一：有没有可读日志

没有 build log，就没有排查基础。客户至少要提供失败部署链接、构建日志、仓库结构或相关截图。只给一句“部署失败”时，你应该先要材料，而不是直接报价。

可以把第一条回复写成标准模板：请提供 Vercel 失败部署链接、最后一次 build log、GitHub 仓库访问方式、最近修改内容、预期上线页面和截止时间。

## 边界二：有没有仓库和配置权限

有些问题只看日志就能判断，有些必须进入仓库或 Vercel 项目设置。你要提前确认自己是否需要只读权限、协作权限，还是由客户按你的建议自行操作。

权限越高，责任越重。新手接单时可以优先做“诊断加指导”，把敏感操作交给客户自己执行。这样既能提供价值，也能降低误操作风险。

## 边界三：是否涉及环境变量

环境变量是部署失败高发点，但也是敏感信息高发点。你可以要求客户提供变量名清单和缺失截图，不要要求客户把真实密钥发给你。

服务范围里要写清楚：你负责判断变量是否缺失、命名是否一致、预览和生产环境是否都配置；客户负责在平台后台填写真实值。

## 边界四：是否需要改代码

有些 build failed 是配置问题，比如 Root Directory 错了；有些是代码问题，比如导入路径错误、类型错误、构建时调用浏览器对象。配置问题适合短平快排查，代码问题要重新评估范围。

当问题从部署配置转成代码修复时，要暂停报价重新确认。不要把“帮你看看部署”一路做成“重构项目”。

## 边界五：是否影响生产服务

如果客户项目已经有真实用户访问，任何重新部署都可能影响线上。你要确认当前是否是 preview、staging 还是 production。预览环境可以更大胆地验证，生产环境要谨慎很多。

涉及主域名、缓存、数据库、支付和用户登录时，不要把任务当作普通静态站上线。需要明确回退方式和验收窗口。

## 风险提醒

接部署排查单时，最大风险是边界漂移。客户一开始说 build failed，后面可能追加域名、数据库、后台接口、SEO、移动端兼容。每新增一类工作，都要重新确认范围。

不要保存客户敏感凭据，不要承诺无法验证的结果，不要在没有授权的情况下修改生产设置。你可以提供清单、诊断、建议和有限修复，但要把风险写清楚。

## 具体步骤

第一步，用询问表收集日志、仓库、权限、环境变量状态和验收目标。

第二步，判断任务属于诊断、配置修复、代码修复还是生产运维。

第三步，给客户写明服务范围：包含哪些检查，不包含哪些操作，需要客户配合什么。

第四步，完成后交付排查报告、修改记录、预览链接和剩余风险。

第五步，把经过客户许可的非敏感经验整理成文章或案例，放到 [工具导航](/tools) 和服务页里，形成搜索流量入口。需要人工协助评估任务边界，可以使用联系入口。
