# Claude Code 新手最容易犯的错误

Claude Code 这类编程助手可以显著提高阅读代码、定位问题和生成修改建议的速度，但它不是“自动交付机器”。新手最常见的问题，不是不会提问，而是太快相信工具输出：没有读仓库结构，没有看 diff，没有跑测试，也没有把客户权限和生产环境风险说清楚。

如果你正准备把 Claude Code 用在 AI 工具实践里，可以先搭配 [ChatGPT、Codex 和 Claude 工作流](/blog/chatgpt-codex-claude-workflow)、[Bug 修复交付说明模板](/blog/bugfix-delivery-note-template)、[/tools/error-explainer](/tools/error-explainer) 和 [/templates](/templates) 使用。目标不是多用工具，而是让每一次修改都有证据。

## 适合谁

适合刚开始用 Claude Code 辅助改代码、解释报错、整理修复说明的新手。你可能能让工具很快生成一段代码，但还不确定它是否符合项目现有结构、是否影响其他页面、是否能通过构建。

也适合准备接低风险代码小单的人，例如小范围样式调整、报错解释、配置文档、测试环境修复、轻量页面问题排查。这些任务可以用清楚的 diff 和测试结果验收。

## 不适合谁

不适合一上来就处理生产数据库、支付后台、客户密钥、真实用户数据或安全漏洞的人。Claude Code 可以帮你看代码，但不能替你承担客户授权、合规判断和生产事故风险。

也不适合不看输出就交付的人。AI 生成的代码可能能运行，也可能只是“看起来合理”。没有人工复核和测试证据，就不应该把它交给客户。

## 具体步骤

第一步，先读项目。让工具总结目录结构、入口文件、相关模块和现有约定。不要一开始就要求“直接修”。你要先知道这个项目用什么框架、脚本、测试方式和代码风格。

第二步，缩小范围。把客户需求拆成一个最小可验证任务：修一个按钮、解释一个报错、调整一个组件、补一段交付说明。范围越小，越容易验证，越适合新手。

第三步，修改后必须看 diff。检查是否只动了目标文件，是否引入无关重构，是否误删内容，是否改动公开状态，是否出现密钥、客户信息或调试日志。

第四步，运行与任务相关的检查。前端项目至少考虑 lint、build、页面手测；内容项目至少考虑内容质量、敏感词、SEO；报错任务至少记录复现方式和修复后结果。

第五步，写交付说明。说明中写清原问题、修改内容、测试结果、未覆盖范围和客户需要确认的事项。不要只写“已完成”。

## 风险提醒

不要把客户完整密钥、后台截图、私密仓库片段或真实用户数据复制进工具。需要调试时，优先使用脱敏日志、测试数据、临时环境和最小必要上下文。

不要让工具替你扩大范围。很多模型会顺手建议重构、升级依赖或改多个模块。新手项目练习时，应该优先完成约定范围，再单独说明额外建议，而不是把所有建议混进本次交付。

## 复盘习惯

每次用 Claude Code 完成任务后，记录四件事：它帮你定位了什么，你人工改了什么，哪些检查通过，哪些风险没有覆盖。几次之后，你会形成自己的提示词和检查流程。

复盘也能帮你判断哪些任务适合接。能复现、能测试、能写清交付的任务更适合新手；需要高权限、真实数据或复杂业务判断的任务，应该先放到待复核列表里。

## 失败信号

如果 Claude Code 连续建议修改很多无关文件，说明任务边界可能没有讲清。此时不要继续接受改动，先回到客户需求，重新写出“本次只解决什么”。如果工具给出的解释和实际报错对不上，也不要继续让它猜，先收集日志、复现步骤和项目脚本。

另一个失败信号是你开始依赖“看起来差不多”。代码交付不能靠感觉。你需要能指出改动文件、修改原因、运行检查和未覆盖范围。如果这些说不出来，就先别交付。

## 客户验收示例

交付时可以写：“本次只修复表单提交后按钮无响应的问题。已在本地测试环境验证点击后进入下一步，`npm run build` 通过。未覆盖生产数据、支付流程和长期监控。请客户按相同路径在预览环境复查。”

这类说明不夸张，但客户能验收。新手用工具项目，最需要训练的就是把 AI 修改变成客户能理解的证据链。
