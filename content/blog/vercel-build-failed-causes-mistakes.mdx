# Vercel build failed 新手常见错误：别一上来就乱改

Vercel build failed 并不可怕，可怕的是新手在没有定位原因前就连续修改配置、删除文件、换包管理器、改环境变量，最后把原本可解释的问题变成无法复盘的混乱现场。部署排查的第一原则是：先看证据，再改东西。

这篇草稿专门写给搜索“Vercel build failed 怎么修”“部署失败哪里错了”的人。想按步骤处理，可以配合 [Vercel build failed 排查清单](/blog/vercel-build-failed-causes-checklist) 使用。

## 适合谁

适合第一次处理 Vercel 构建失败的人，尤其是本地能跑、线上失败、日志很多看不懂的情况。你不需要马上成为部署专家，但需要避免几个高频误区。

也适合准备接部署排查小单的人。很多客户需要的不是你炫技，而是你能稳定收集材料、说明风险、给出可验证结果。

## 不适合谁

不适合想把 AI 输出直接当最终方案的人。AI 可以解释日志，但它看不到客户账号里的真实配置，也不知道你刚刚改过哪些设置。

如果你没有仓库权限、没有构建日志、没有客户授权，却准备直接操作生产项目，这篇不建议你继续。先把材料补齐，再决定是否接手。

## 错误一：只看最后一行 failed

很多构建日志最后都会出现 failed，但真正原因通常在更前面。你要找第一处具体报错，例如某个包安装失败、某个文件导入不存在、某个变量缺失、某段 TypeScript 检查没通过。只看最后一行，容易把结果当原因。

正确做法是从日志中搜索 error、failed、missing、undefined、cannot find、invalid 等关键词，再回到报错前后十几行看上下文。每次定位都要复制文件路径和命令，不要只写“报错了”。

## 错误二：随手删除 lockfile

看到依赖安装失败，有些人会立刻删除 lockfile 或换包管理器。这很危险，因为 lockfile 记录了项目原本依赖版本。你删掉它，可能短期避开一个安装问题，却引入一堆版本变化。

更稳的方法是先确认项目原本使用 `npm`、`pnpm` 还是 `yarn`。如果仓库里同时存在多个 lockfile，要和客户确认保留哪一个，再提交清晰说明。不要在没有确认的情况下把所有依赖重新生成一遍。

## 错误三：忽略环境变量

本地能跑线上失败，很常见的原因是 `.env.local` 没有同步到 Vercel。新手容易盯着代码改半天，却忘了线上根本没有数据库地址、API Key 或公开配置。

注意，排查环境变量不等于收集客户密钥。你要确认变量名、用途和是否已配置，让客户在平台后台填写敏感值。记录里只写变量名和状态，不保存真实内容。

## 错误四：把根目录看错

很多项目不是仓库根目录直接部署，而是放在 `apps/web`、`frontend`、`site` 之类的子目录。Vercel 如果从错误目录构建，就会找不到脚本、找不到依赖或输出错误页面。

遇到 monorepo，要先画出仓库结构，再确认 Vercel 的 Root Directory。这个动作很普通，但能解决大量“我明明有 package.json，平台却说找不到”的问题。

## 错误五：没有交付记录

部署排查完成后，如果你只说“好了”，客户很难信任，也不方便后续维护。更糟的是，过几天同样问题出现，你也想不起来当时改了什么。

交付记录至少包括失败原因、修改项、验证链接、仍需客户确认的内容和下一步建议。它既是服务质量，也是你后续做案例、写文章、形成流量入口的素材。

## 风险提醒

不要在客户生产项目里做无法回退的尝试。删除环境变量、改域名解析、切换框架输出、升级大量依赖，都可能影响线上服务。每次改动前要确认授权和回退方式。

如果日志涉及数据库迁移、用户数据、支付配置、账号权限或安全漏洞，新手不要单独处理。你可以做材料整理和初步判断，但关键决策要交给客户或资深开发者。

## 具体步骤

第一步，打开失败部署日志，找到第一处具体报错。

第二步，确认失败阶段：依赖安装、生产构建、输出生成、路由访问，还是域名配置。

第三步，检查包管理器、lockfile、build script、Root Directory、Node 版本和环境变量。

第四步，只改一项后重新部署，并记录新的结果。

第五步，把排查过程整理成报告。需要模板时，可以使用 [工具导航](/tools) 里的下载入口，或联系人工协助把聊天记录整理成客户能看懂的说明。
