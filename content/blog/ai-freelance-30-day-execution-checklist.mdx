# 新手 30 天 AI 工具实践执行检查清单

先给结论：30 天 AI 工具实践计划要靠检查清单执行，而不是靠热情硬撑。每天记录一个小任务，每周整理一个可展示结果，月底再判断是否适合开始投低风险项目。

这份清单适合复制到表格、Notion 或纸质笔记里。重点不是每天做很多事，而是每天留下证据：你学了什么、做了什么、验证了什么、下一步是什么。

## 适合谁

适合刚开始学习 AI 工具实践的新手，尤其是还没有作品集、没有交付记录、没有固定工具流程的人。

也适合已经试过几天但容易中断的人。清单能把“今天不知道做什么”的问题拆成很小的动作。

## 不适合谁

不适合想跳过练习直接承诺复杂项目的人。30 天计划是准备流程，不是结果保证。

也不适合想用自动化替自己投标或公开大量未审核内容的人。早期更应该保守：草稿保持 draft，Proposal 保持初稿，对外动作人工确认。

## 具体步骤

按 4 个阶段执行：

1. 第 1 周：工具环境和基础练习。
2. 第 2 周：低风险小任务练习。
3. 第 3 周：作品集和交付记录整理。
4. 第 4 周：项目判断和 Proposal 初稿练习。

每个阶段都要有产出。没有产出，就回到更小的任务，不要继续堆新工具。

## 第 1 周检查

```text
工具环境：
1. 本地 Node 环境可用。
2. GitHub 账号可用。
3. 至少会创建一个简单仓库。
4. 至少会运行 npm install 和 npm run build。
5. 至少会使用一个 AI 编程工具解释报错。
6. 至少部署过一个测试页面。
7. 知道 API Key、密码、Token 不能随便输入给工具。
```

第 1 周不追求承接项目，只追求工具能跑通。

## 第 2 周检查

```text
低风险练习：
1. 完成一个静态页面修改。
2. 完成一次 CSS 样式调整。
3. 完成一次构建报错解释。
4. 完成一次文案或 Proposal 初稿整理。
5. 每个练习都有截图或链接。
6. 每个练习都有一句复盘。
7. 没有接触客户真实敏感数据。
```

第 2 周的目标是形成“小任务可完成”的证据，不是证明自己能做复杂项目。

## 第 3 周检查

```text
作品集整理：
1. 选择 2-3 个练习作为案例。
2. 每个案例写清问题、工具、步骤、验证和结果。
3. 每个案例说明限制和未覆盖范围。
4. 不伪造客户案例。
5. 不使用未授权截图。
6. 不把学习 demo 包装成真实商业交付。
7. 保存案例链接或截图。
```

作品集要真实。小而清楚的案例，比夸大的案例更可信。

## 第 4 周检查

```text
项目判断：
1. 每天找 3 个需求练习判断。
2. 标记适合投、需要确认、暂时不接。
3. 写出需要问客户的问题。
4. 写出交付包含和不包含。
5. 用 Proposal 生成器做初稿。
6. 人工删除夸大承诺。
7. 不投高风险项目。
```

第 4 周可以练 Proposal，但不必急着大量投。先把判断能力练稳。

## 每日执行表

```text
日期：
今日阶段：第 __ 周
今日任务：

使用工具：

完成结果：

验证方式：

遇到的问题：

明天继续：

是否涉及敏感数据：是 / 否
是否需要人工复核：是 / 否
```

每天填完这张表，就算当天只完成一个很小的动作，也是在推进。

## 每周复盘表

```text
本周完成：
1.
2.
3.

可展示结果：
1.
2.

最大问题：

下周要减少的动作：

下周要加强的动作：

是否有不适合继续公开或投标的内容：
```

复盘时要删掉无效动作。比如每天换工具、看太多教程、没有产出，都要减少。

## 常见错误

第一个错误是清单太满。每天 10 个任务很容易放弃。每天只要一个明确任务就够。

第二个错误是只学不做。教程看完没有截图、链接和复盘，就很难变成作品集。

第三个错误是忽略风险。看到项目就投，不先判断是否涉及生产环境、支付、隐私和账号权限。

第四个错误是没有人工审核。AI 生成的内容、代码和 Proposal 初稿，都要人工确认后才能对外使用。

## 风险提醒

不要承诺收入结果，不要承诺一定成交，不要把 30 天计划当作职业保证。遇到平台外结算请求、免费完整测试、高权限账号、生产数据库、支付或隐私数据时，先暂停。

所有公开文章、客户回复和真实交付都需要人工审核。练习可以快，交付要稳。

## 总结

30 天执行清单的作用，是把模糊目标变成每天可验证的小动作。先跑通工具，再做低风险练习，再整理作品集，最后练项目判断和 Proposal。每一步都留记录，才不会把学习变成空转。
