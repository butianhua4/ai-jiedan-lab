# GitHub 常用命令表新手怎么记

新手学 GitHub 命令时，最容易犯的错是背命令表，却不知道什么时候用。更稳的方式是按场景记：我现在要看状态、同步代码、保存改动、提交记录、推到远程，还是检查自己改了什么。

严格说，很多命令属于 Git，不是 GitHub 本身。Git 是本地版本控制工具，GitHub 是托管仓库和协作平台。新手不需要一开始把概念讲得很复杂，但要知道：命令会改变本地文件和提交历史，远程权限则可能影响团队仓库。

## 适合谁

适合正在做个人练习项目、作品集网页、AI 辅助改代码或接低风险小任务的新手。你可能只会复制命令，但还不清楚每一步在保存什么。

也适合准备把 Codex、Cursor 或其他 AI 工具生成的改动提交到仓库的人。AI 可以帮你改文件，但提交之前仍要人工看 `git status` 和 `git diff`。

## 不适合谁

不适合直接在客户生产仓库里练命令的人。客户仓库可能有多人协作、分支保护、CI 检查和发布流程，新手不应该拿不熟的命令试错。

也不适合一上来学习改历史、强制推送、删除分支等高风险命令。先把状态、差异、暂存、提交、推送这些基本动作练熟。

## 风险提醒

不要在没看清当前状态时提交，不要把密钥、客户数据、个人 token、`.env` 文件或私密截图放进仓库。提交前先看 `git diff`，确认每个改动都能解释。

如果任务需要客户添加协作者、授权仓库访问、创建远程仓库或修改保护分支规则，把它写成客户侧确认事项或明日待办。没有权限时，不要假装已经能推送。

## 具体步骤

第一步，只在自己的练习仓库里练。新建一个小项目，故意改一个 README 或页面文案。

第二步，先运行 `git status`，看 Git 认为哪些文件变了。

第三步，用 `git diff` 看具体改动。不要只看文件名，要看内容。

第四步，用 `git add` 暂存你确认要提交的文件。

第五步，用 `git commit` 写一条清楚的提交消息。

第六步，如果远程仓库是你自己的，再用 `git push` 推送。

第七步，推送后去 GitHub 页面确认提交记录。不要只相信命令行显示，要学会回到网页核对。

## 最常用的命令

```bash
git status
git diff
git add README.md
git commit -m "Update README"
git push
```

这五个命令足够覆盖最小练习流程。新手先把它们练熟，比背几十个命令更有用。

`git status` 是你的安全带。每次改动前、提交前、提交后都可以看一眼。它会告诉你哪些文件已修改、哪些文件已暂存、当前分支是否领先或落后。

`git diff` 是你的复核工具。它告诉你具体改了什么。做项目时，所有不能解释的 diff 都应该暂停处理。

## 同步远程代码

常见命令是：

```bash
git pull
```

但新手不要机械执行。先看当前状态，如果本地有未提交改动，直接拉取可能产生冲突。更稳的流程是：先 `git status`，确认本地干净，或者先把自己的改动提交到本地，再考虑同步远程。

如果这是客户或团队仓库，拉取前最好确认你在正确分支。不要在不确定分支时随便改文件。

## 分支怎么理解

分支可以理解成一条独立工作线。新手接小任务时，常见做法是从主分支拉一个新分支，在新分支上修改，再提交 PR。

常见命令：

```bash
git branch
git switch -c fix-mobile-button
```

如果项目已有协作规范，按项目规范来。不要自己随便命名分支、直接推主分支或绕开审核流程。

## 提交消息怎么写

提交消息不需要文学感，只要说明做了什么。比如：

```bash
git commit -m "Fix mobile button overflow"
git commit -m "Update pricing table copy"
git commit -m "Document npm install error steps"
```

不要写 `fix`、`update`、`final` 这种太模糊的消息。以后复盘时，你会看不懂自己当时做了什么。

## AI 改动怎么提交

AI 工具改完代码后，不要立即 `git add .`。先用 `git status` 看文件范围，再用 `git diff` 看内容。如果 AI 改了无关文件、格式化了整片代码或删除了你不理解的逻辑，要先停下来。

更安全的做法是只暂存目标文件，例如：

```bash
git add content/blog/example.mdx
```

这样能降低误提交无关内容的风险。

## 项目前怎么练

第一天练 `status` 和 `diff`。第二天练 `add` 和 `commit`。第三天练 `push` 到自己的测试仓库。第四天练创建分支。第五天练写一条清楚的提交说明和 PR 描述。

练习时把每次命令和结果写进复盘。能复述清楚命令含义，才适合把 Git 用到客户项目里。
