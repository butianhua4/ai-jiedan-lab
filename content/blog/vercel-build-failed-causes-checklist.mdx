# Vercel build failed 排查清单：从日志到重新部署

遇到 Vercel build failed，最有用的不是一句“帮我看看”，而是一份可执行清单。清单的价值在于让你按顺序排除问题，不遗漏关键材料，也方便把排查过程交给客户复核。搜索这类问题的人通常已经卡住了，所以文章要直接给步骤、记录表和判断边界。

这篇草稿用于承接“Vercel 部署失败怎么排查”“build failed 看哪里”“本地能跑线上失败”等搜索需求。你可以先读主教程 [Vercel build failed 常见原因](/blog/vercel-build-failed-causes)，再用这篇做操作记录。

## 适合谁

适合已经拿到 Vercel 失败链接、GitHub 仓库或构建日志的人。你不一定要完全懂前端框架，但要能复制日志、识别文件名、确认命令，并把每一步结果写下来。

也适合 AI 工具新手做售前判断。客户发来问题后，你可以先按清单要材料，再判断任务是不是小范围诊断。如果材料不完整，就不要直接报价。

## 不适合谁

不适合没有权限、没有日志、没有仓库结构，却想直接给结论的人。部署失败的原因可能差一行配置，也可能牵涉到生产账号，不能靠猜。

如果客户不愿提供任何可验证材料，只要求你马上承诺修好，这类任务不适合新手接。你可以礼貌说明需要 build log、预览链接、仓库结构和验收标准。

## 清单一：先收集材料

需要记录项目名称、仓库地址、失败部署链接、失败时间、最近一次改动、使用的框架、包管理器、构建命令、Root Directory、是否有环境变量、是否是预览环境还是生产环境。每一项都要写成可复制的文字，而不是只存在聊天记录里。

如果客户只给截图，你要继续要原始日志。截图可以帮助定位，但长日志里的第一处报错、文件路径和命令输出更重要。做项目时可以把“请提供 Vercel build log 和仓库结构截图”作为第一条标准回复。

## 清单二：确认失败阶段

把失败阶段分成安装依赖、运行构建、生成输出、部署路由、访问页面五类。安装依赖失败通常和 lockfile、包版本、私有包权限有关；运行构建失败可能是 TypeScript、导入路径、框架配置、环境变量；部署后页面打不开则可能是路由、输出目录或域名问题。

不要把所有失败都叫“部署失败”。你越能说清楚失败阶段，客户越容易理解工作范围，也越容易判断后续是不是需要开发改代码。

## 清单三：本地复现

在本地先执行安装命令，再执行生产构建命令。如果项目使用 `pnpm`，就不要临时改成 `npm`；如果仓库只有 `package-lock.json`，就不要随手生成新的 lockfile。复现时记录命令、Node 版本、报错文件和结果。

本地复现失败，通常说明项目自身存在问题。线上失败但本地成功，通常要重点检查 Vercel 项目设置、环境变量和平台环境差异。这个判断能帮你避免在错误方向上消耗时间。

## 清单四：检查配置

检查 `package.json` 里的 scripts，确认有明确 build 命令。检查 Root Directory 是否指向真正项目目录。检查 Vercel 的 Framework Preset、Build Command、Install Command、Output Directory 是否和项目匹配。检查 Node 版本是否有项目要求，例如 `.nvmrc` 或 engines 字段。

环境变量要单独列成表：变量名、用途、是否敏感、预览环境是否配置、生产环境是否配置、谁负责填值。不要在文章、工单或聊天里保存真实密钥。

## 风险提醒

清单不是让你把所有配置都改一遍。排查时每次只改一个方向，并保存修改前后的状态。尤其是生产项目、客户主域名、数据库地址、支付相关配置和账号权限，任何改动都要先确认授权。

AI 可以帮你总结日志，但不能替代人工判断。日志里出现安全、隐私、支付、数据库迁移等内容时，应当升级给有经验的人处理。

## 具体步骤

第一步，把失败日志复制出来，标记第一处错误和关联文件。

第二步，运行本地生产构建，判断是否可复现。

第三步，对照依赖、命令、根目录、环境变量、Node 版本逐项填写结论。

第四步，根据结论只修改一项配置或一处代码，然后重新部署。

第五步，访问预览链接，检查首页、关键路由、静态资源、移动端和 sitemap。需要更完整模板时，可以从 [工具导航](/tools) 下载记录表，或联系人工协助整理排查报告。
